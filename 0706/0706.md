### 任務 1：案例拆解

以小組為單位，針對「社團活動報名與場地申請系統」完成下列分析：

1. 列出至少 5 個利害關係人（stakeholders）。
2. 列出至少 3 個目前流程中的痛點（pain points）。
3. 列出至少 5 種系統需要保存的資料（data）。
4. 列出至少 3 條業務規則（business rules）或限制。
5. 列出至少 2 個可能造成系統失敗的風險（risks）。

完成標準：

- 痛點（pain point）需要描述目前問題，不是直接寫解法。

### 任務 2：候選題目發想

每組提出 2-3 個候選題目。每個題目需包含：

1. 系統名稱。
2. 目標使用者。
3. 要解決的問題。
4. 三個核心任務。
5. 系統不處理的範圍。

輸出格式：

```
候選題目 1：
系統名稱：
目標使用者：
要解決的問題：
核心任務：
1.
2.
3.
本期不處理：
```

完成標準：

- 至少 2 個候選題目。
- 每個題目都要能在 5 週內做出原型（prototype）。
- 題目不能只描述技術，例如「人工智慧系統（AI system）」，必須描述使用情境。

### 任務 4：撰寫問題陳述（problem statement）

使用句型完成問題陳述（problem statement）：

```
對於 [目標使用者（target users）] 而言，目前在 [目前情境（current situation）] 中，常發生 [痛點與影響（pain / impact）]。
本專案希望設計一個 [系統概念（system concept）]，協助 [期待改善（desired change）]，並在本期聚焦於 [範圍焦點（scope focus）]。
```

完成標準：

- 至少 120 字。
- 必須包含目標使用者（target users）、目前情境（current situation）、痛點與影響（pain / impact）、期待改善（desired change）、範圍焦點（scope focus）。
- 不可以只列功能。

### 任務 5：建立範圍陳述（scope statement）

每組完成：

1. 專案目標（project goal）：1-2 句。
2. 範圍內項目（in-scope）：至少 5 項。
3. 範圍外項目（out-of-scope）：至少 5 項。
4. 核心任務（core tasks）：至少 3 個可展示任務。
5. 初步成功標準（initial success criteria）：至少 3 項。

### 可用提示詞範例（Prompt Examples）

題目診斷：

```
請幫我診斷以下系統分析與設計課程專案題目是否適合 5 週暑修課。
請檢查：
1. 目標使用者（target users）是否明確
2. 是否有至少 3 個核心任務
3. 是否有資料需要管理
4. 是否有角色、權限或規則
5. 是否能做出原型（prototype）展示
6. 範圍（scope）是否過大或過小

請不要直接幫我完成作業，只提出診斷與追問問題。

題目：
[貼上題目]
```

問題陳述檢查（Problem statement check）：

```
請檢查以下問題陳述（problem statement） 是否符合系統分析與設計課程要求。
請指出缺少哪些元素：
1. 目標使用者（target users）
2. 目前情境（current situation）
3. 痛點與影響（pain / impact）
4. 期待改善（desired change）
5. 範圍焦點（scope focus）

請給我修改建議，但不要直接代寫完整答案。

問題陳述（Problem statement）：
[貼上內容]
```

範圍風險檢查（Scope risk check）：

```
以下是我們的範圍內項目（in-scope）和範圍外項目（out-of-scope）。
請幫我判斷是否有範圍蔓延（scope creep）風險，並指出哪些項目可能太大、太小或不適合 5 週原型（prototype）。

範圍內項目（In-scope）：
[貼上內容]

範圍外項目（Out-of-scope）：
[貼上內容]
```

不合格使用：

1. 只貼上生成式 AI（generative AI）產生的完整專案章程（Project Charter）。
2. 無法說明生成式 AI（generative AI）產出哪些地方被修改。
3. 生成式 AI（generative AI）建議與小組情境不一致，但仍照抄。
4. 生成式 AI（generative AI）幫忙捏造不存在的訪談或使用者資料。

## 常見錯誤與診斷

| 常見錯誤 | 診斷問題 | 修正方式 |
|---|---|---|
| 題目太大 | 是否需要正式部署、金流、跨系統串接？ | 把正式整合放到範圍外（out-of-scope），用原型（prototype）模擬 |
| 題目太小 | 是否只有單一新增、查詢、修改、刪除（CRUD）畫面？ | 加入至少 3 個核心任務與角色互動 |
| 題目太抽象 | 是否只有「改善」、「管理」、「平台」等詞？ | 加入具體使用者、情境、流程與資料 |
| 使用者不明 | 是否只寫「使用者」？ | 改成具體角色，例如學生、老師、行政人員、社團幹部 |
| 技術先行 | 是否一開始就說要用生成式 AI（generative AI）、應用程式（App）、儀表板（Dashboard）？ | 改成先描述問題，再決定技術是否必要 |
| 問題陳述（problem statement）只有功能 | 是否看得出誰遇到什麼問題？ | 加入目標使用者（target users）、目前情境（current situation）、痛點與影響（pain / impact） |
| 沒有驗收標準 | 是否無法判斷期末完成？ | 寫出初步成功標準（initial success criteria） |

## 形成性評量

### 檢核 1：概念題

請用一句話回答：

```
系統分析與設計為什麼不能從畫使用者介面（UI）開始？
```

可接受答案要包含：問題、使用者、流程、資料、規則、需求或驗收標準其中至少兩個元素。

### 檢核 2：辨識題

下列哪一個比較像可分析的專案題目？

1. 做一個 人工智慧校園系統（AI campus system）。
2. 做一個好用的社團網站。
3. 面向社團幹部與課外組的活動報名、候補與場地申請流程整合系統。

標準答案：3。理由是它有目標使用者、流程與系統範圍。

### 檢核 3：一句話說明

請每組用一句話說明：

```
我們的系統要幫助誰，在什麼情境下，解決什麼問題？
```

若一句話中沒有具體使用者與情境，需回去重寫。

### 檢核 4：離場回饋（Exit Ticket）

下課前每組繳交：

1. 選定題目。
2. 問題陳述（problem statement）。
3. 範圍內項目（in-scope）5 項。
4. 範圍外項目（out-of-scope）5 項。
5. 3 個核心任務。
6. 生成式 AI 使用紀錄（generative AI use log）。
