## 學生任務

### 任務 1：案例拆解

以小組為單位，針對「社團活動報名與場地申請系統」完成下列分析：
1. 列出至少 5 個利害關係人（stakeholders）。
2. 列出至少 3 個目前流程中的痛點（pain points）。
3. 列出至少 5 種系統需要保存的資料（data）。
4. 列出至少 3 條業務規則（business rules）或限制。
5. 列出至少 2 個可能造成系統失敗的風險（risks）。

輸出格式：
```
小組名稱：TKU_114-3_SAD_Group04

利害關係人（Stakeholders）：
1. 學生社員（想要報名社團活動的學生）
2. 社團幹部（負責建立活動、審核報名名單的社團負責人／總務）
3. 學務處課外活動組承辦人員（負責審核場地申請與活動合規性）
4. 總務處場地管理人員（負責維護場地資訊、確認場地是否可用、處理場地衝突）
5. 系統管理員（負責維護系統運作、帳號權限與資料備份）

痛點（Pain Points）：
1. 目前報名多以紙本表單或Google表單填寫，社團幹部需手動彙整報名者名單，容易漏登或重複登記，且無法即時掌握剩餘名額。
2. 場地申請需分別向總務處與學務處遞交紙本申請單，兩邊資訊可能不同步，導致曾經發生同一時段被兩個社團同時申請、到了現場才發現場地衝突的情況。
3. 活動異動（EX：時間更改、取消）僅透過口頭或社群通知，導致部分已報名學生未能即時收到更新資訊，到場後才發現活動取消或改期。
4. 場地審核進度不透明，社團幹部須主動打電話或跑公文詢問審核結果，等待時間長且無法追蹤申請狀態。
5. 報名資料（EX：緊急聯絡人、餐飲需求、是否為社員）分散在不同表單或紙本上，活動結束後難以統計與留存，造成下學期規劃時無歷史資料可參考。

資料（Data）：
1. 使用者帳號資料：學號、姓名、系級、聯絡方式、角色（學生／幹部／承辦人員）
2. 社團資料：社團名稱、指導老師、社團類別、現任幹部名單
3. 活動資料：活動名稱、活動時間（起訖）、活動描述、主辦社團、報名人數上限、報名截止時間
4. 場地資料：場地名稱、容納人數上限、可用設備（投影機、麥克風等）、可申請時段、場地管理單位
5. 報名紀錄：報名者學號、報名活動ID、報名時間、報名狀態（已確認／候補／取消）
6. 場地申請紀錄：申請社團、申請場地、申請時段、審核狀態（待審／通過／退回）、審核意見、審核人員

業務規則（Business Rules）：
1. 同一場地在同一時段僅能被一個活動核准使用，系統須在申請時自動檢查時段是否衝突，衝突時不得同時核准。
2. 活動報名人數不得超過該場地核准之容納人數上限，達到上限後系統應自動關閉報名或轉入候補名單。
3. 場地申請須於活動預定日前至少7天送出，逾期申請系統應提示不受理或標示為「加急申請」需額外簽核。
4. 非本校在籍學生（含校友、外賓）報名須經社團幹部人工審核通過後才能取得正式報名資格，不可直接自動確認。
5. 場地申請經審核退回後，社團須修改申請內容重新送出，不可沿用原申請單直接覆核。

風險（Risks）：
1. 若場地衝突檢查邏輯未正確實作或資料同步延遲，可能導致兩個社團在同一時段、同一場地被同時核准，造成活動當天現場衝突與資源搶用。
2. 系統若無妥善的權限控管（EX：一般學生可修改審核狀態），可能導致未經授核的活動被誤標為「已核准」，影響場地使用秩序與校方管理責任。
3. 尖峰時段（EX：學期初社團博覽會、多社團同時搶熱門場地）大量申請同時湧入，系統若無承載測試，可能發生資料寫入衝突或系統當機，造成申請資料遺失。
4. 系統若未定期備份報名與審核資料，一旦伺服器故障或資料庫損毀，將導致當學期活動與場地紀錄全數遺失，且無法回溯歷史資料。
```

完成標準：
- 每一欄都要能對應案例，不可只寫空泛詞。
- 利害關係人（stakeholder）不能只寫「使用者」，要具體。
- 痛點（pain point）需要描述目前問題，不是直接寫解法。

### 任務 2：候選題目發想

每組提出 2-3 個候選題目。每個題目需包含：
1. 系統名稱。
2. 目標使用者。
3. 要解決的問題。
4. 三個核心任務。
5. 系統不處理的範圍。

輸出格式：
```
候選題目 1：
系統名稱：基於結構化查詢的校園二手教科書媒合與面交管理平台
目標使用者：買家學生（想省錢的買家）、賣家學生（想清書櫃的賣家）
要解決的問題：校內二手書買賣依賴Line社群，同學刊登格式不一致（EX：書名英文/中文縮寫混用），導致關鍵字搜尋困難、訊息洗版；且缺乏交易狀態管理，常發生已售出未下架，以及面交遭放鳥卻無法處置的情形。
核心任務：
1. 賣家結構化刊登與買家精準搜尋
2. 面交邀約流程與鎖定書籍狀態
3. 面交完成與結案紀錄
本期不處理：線上金流支付（EX：Line Pay/刷卡）、物流寄送、即時線上聊天室、ISBN全球資料庫串接、非教科書之一般雜物市集

候選題目 2：
系統名稱：校園運動場地與球隊約戰系統
目標使用者：球隊隊長、一般運動學生
要解決的問題：校園運動場地與球隊媒合缺乏系統化管理，也缺乏即時場地使用狀態（EX：學生到場才發現場地已被占用或被同時預約，白跑一趟）；且缺乏統一約戰媒合管道，系隊隊長只能私下詢問其他系隊，導致約戰效率低下、資訊不透明，也容易流於單純聊天室。
核心任務：
1. 場地預約與場地即時狀態
2. 發起約戰
3. 球隊加入紀錄
本期不處理：與學校體育處正式官方場地借用系統的後端資料庫連動、球賽現場裁判與記分板即時串流功能、約戰球隊之不當言論即時線上聊天室
```

完成標準：
- 至少 2 個候選題目。
- 每個題目都要能在 5 週內做出原型（prototype）。
- 題目不能只描述技術，例如「人工智慧系統（AI system）」，必須描述使用情境。

### 任務 3：候選題目評選

每組從候選題目中選出 1 個主題。使用下列表格評估：

| 評估項目 | 候選題目 1：基於結構化查詢的校園二手教科書媒合與面交管理平台 | 候選題目 2：校園運動場地與球隊約戰系統 |
| -------- | -------------------------------------- | ----------------------------------- |
| 使用者是否明確 | 是：買書的學生、賣書的學生（皆為校內學生） | 尚可：球隊隊長、一般運動學生 |
| 是否有 3 個以上核心任務 | 有：書籍規範刊登、精準關鍵字／ISBN查詢、預約面交狀態變更 | 有：場地預約、發起約戰、球隊加入 |
| 是否有資料需要管理 | 有：書籍主檔、課程對應欄位、面交訂單狀態 | 有：場地時段、球隊隊員、約戰紀錄 |
| 是否有角色、權限或規則 | 是：強制格式填寫規則、逾期自動下架、放鳥停權規則 | 尚可：場地衝突阻擋規則 |
| 是否能在 5 週內做原型（prototype） | 是：聚焦在資料的「結構化輸入與查詢」 | 尚可：場地時間切分與球隊審核邏輯較繁瑣 |
| 是否容易訪談或模擬需求 | 是：全班都有用Line社群買賣二手書被格式氣到的經驗 | 尚可：只有有運動習慣的同學較能理解 |

最後選定題目：
```
基於結構化查詢的校園二手教科書媒合與面交管理平台
```

選擇理由：
```
1. 切中現有痛點：目前學校的二手書買賣依賴Line社群，但因為同學刊登書籍相關資料的方式皆不一致，導致極難以關鍵字搜尋，資訊極度混亂。
2. 訪談與驗證極其容易：使用者全部都是身邊的同學，不需要特地去拜託校外商家合作，並且5週暑修內完成需求擷取的可行性最高。
3. SA建模價值高：透過規範賣家必須輸入「科目名稱」、「開課老師」、「ISBN」等結構化資料，能夠展現出「資料管理與欄位設計」的核心能力，並且能透過系統規則（EX：逾期放鳥停權）來收斂範圍。
```

完成標準：
- 至少比較 2 個候選題目。
- 最後選出 1 個主題，並寫出選擇理由。

### 任務 4：撰寫問題陳述（problem statement）

使用句型完成問題陳述（problem statement）：
```
對於每學期有買賣二手書需求的校內學生而言，目前在依賴的Line交易社群缺乏統一格式的情境中，常發生同學刊登的書名、版本、授課老師等資訊往往因人而異（EX：同一本資訊系統分析書名有人打英文、有人打中文縮寫），導致買家學生極難透過關鍵字精準查詢，訊息也容易被刷屏洗版；此外，由於缺乏結構化的交易狀態管理，經常發生賣家書籍已售出卻未更新或刪文，以及買賣雙方在約定面交時遭受惡意放鳥卻無法處置的情形。

本專案希望設計一個針對校園二手書刊登與面交情境的結構化管理平台，協助改善資訊不對稱與搜尋效率，透過強制結構化欄位（EX：科目名稱、開課老師、ISBN）確保資料齊全且格式一致，並提供清晰的交易狀態流轉與放鳥停權機制，並在本期聚焦於校園實體面交場景下的資料結構化管理。
本期不處理：真實線上刷卡或Line Pay金流、超商跨校物流、買賣雙方即時線上聊天室功能。
```

完成標準：
- 至少 120 字。
- 必須包含目標使用者（target users）、目前情境（current situation）、痛點與影響（pain / impact）、期待改善（desired change）、範圍焦點（scope focus）。
- 不可以只列功能。

### 任務 5：建立範圍陳述（scope statement）

每組完成：
1. 專案目標（project goal）：1-2 句。
2. 範圍內項目（in-scope）：至少 5 項。
3. 範圍外項目（out-of-scope）：至少 5 項。
4. 核心任務（core tasks）：至少 3 個可展示任務。
5. 初步成功標準（initial success criteria）：至少 3 項。

```
1. 專案目標（project goal）：開發一套校園二手教科書結構化媒合管理平台，透過標準化欄位與精準篩選功能解決現有社群查詢不易的痛點，並建立透明的面交預約追蹤機制，以此提升校園二手書流轉的效率與交易誠信。
2. 範圍內項目（in-scope）：
    (1). 結構化二手書上架：賣家上架時必須填寫標準欄位（科目名稱、開課老師、書籍版本、ISBN、書況、價格、預期面交地點）並上傳書籍照片。
    (2). 多條件精準搜尋篩選：買家可透過關鍵字（EX：科目名稱、開課老師、ISBN）進行模糊查詢，並可依價格區間進行篩選。
    (3). 面交邀約流程：買家選定書籍後點擊「預約購買」，系統會將書籍狀態切換為「已預約」，並自動產生面交確認單。
    (4). 面交進度狀態管理：提供雙方確認介面，由賣家於現場面交收現後點擊「確認完成交易」，書籍狀態正式流轉為「已售出」。
    (5). 違規放鳥檢舉停權：如若面交遭受放鳥，買賣任一方可於系統按下「放鳥回報」，經累計達2次者，系統自動予以停權處分。
3. 範圍外項目（out-of-scope）：
    (1). 第三方線上金流支付系統：為了在 5 週內收斂，系統不經手任何金流，一律採現場面交時「現金結帳」或當面轉帳。
    (2). 物流寄送與店到店系統：本期僅支援校園實體面交（EX：商管大樓、圖書館前），不處理郵寄或超商代收付。
    (3). 即時線上聊天室：不開發像蝦皮那樣的即時通訊功能，雙方聯繫一律透過系統內建的面交時間地點約定欄位。
    (4). ISBN全球資料庫PI串接：不串接外部圖書館資料庫來自動抓取書名，ISBN欄位在本期純作為資料庫內的比對與搜尋索引。
    (5). 非教科書之一般雜物市集管理：嚴格限制商品分類，本期不處理衣服、電子產品等一般校園二手物買賣。
4. 核心任務（core tasks）：
    (1). 賣家結構化刊登與買家精準搜尋
    (2). 面交邀約流程與鎖定書籍狀態
    (3). 面交完成與結案紀錄
5. 初步成功標準（initial success criteria）：
    (1). 書籍成功上架且狀態為「上架中」；搜尋功能成功篩選，並正確列出符合條件之項目
    (2). 產生一筆內含面交資訊的媒合紀錄；該書籍在平台上暫時隱藏或顯示為不可點選
    (3). 書籍狀態成功變更為「已售出」；雙方歷史紀錄中皆能查詢到此筆成功交易
```
