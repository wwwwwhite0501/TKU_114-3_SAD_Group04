最後選定題目：

```
診所候診叫號與現場預約系統
```

選擇理由：

```
1. 角色明確：病患與櫃檯人員兩種角色清楚、易懂，不像題目2的角色身分重疊、界線模糊。
2. 任務類型完整且單一：新增、查詢、更新狀態三種操作皆有涉及，且不涉及金流或協商機制，複雜度可控。
3. 風險容易收斂：只要限定「單一醫生單一隊列」「查詢採手動刷新」，即可將範圍鎖定在5週內完成；題目2的協商與身分驗證需求較難靠排除範圍完全解決。
4. 訪談容易：看診候診經驗人人都有，容易找人問到具體流程與痛點。
```

### D. 目標使用者（Target Users）

```
| 角色 | 他/她在系統中的任務 | 目前痛點 |
|---|---|---|
| 現場候診病患 | 報到後透過看診號查詢目前叫號進度（目前看診到第幾號、前面還有幾人） | 報到後不知道還要等多久，只能在候診區乾等，或反覆詢問櫃檯 |
| 櫃檯掛號人員 | 將現場報到的病患加入候診隊列，標記看診醫生與預估看診時間；調整隊列（插入急診病患、標記爽約取消號次） | 需手動翻紙本或口頭喊號叫病人，遇到插隊、臨時取消號次時容易造成秩序混亂與病患抱怨 |
| 看診醫生 | 依照系統顯示的隊列順序看診，掌握下一位候診病患資訊 | 不清楚候診區目前實際等候人數與狀況，需靠櫃檯口頭告知才能掌握進度 |
| 診所負責人／管理人員 | 檢視每日隊列使用狀況（如看診人數、爽約次數），評估現場叫號流程是否順暢 | 目前無法量化候診等待時間或爽約狀況，難以檢討現場流程或人力安排是否需調整 |
```

### E. 問題陳述（Problem Statement）

```
小型診所的候診叫號流程目前仍仰賴人工作業。病患報到後不知道還要等多久，只能在候診區乾等或反覆詢問櫃檯；櫃檯人員需手動翻紙本、口頭喊號，遇到插隊或爽約時容易造成秩序混亂與抱怨；醫生也無法即時掌握候診區的等候狀況；診所管理人員則缺乏看診人數、爽約次數等數據，難以檢討流程或調整人力安排。
```
