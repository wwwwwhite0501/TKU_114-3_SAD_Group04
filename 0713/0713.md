#### 1. 專案背景

```
對於每學期有買賣二手書需求的校內學生而言，目前依賴的Line交易社群缺乏統一格式，同學刊登的書名、版本、授課老師等資訊往往因人而異（EX：同一本資訊系統分析書名有人打英文、有人打中文縮寫），導致買家學生極難透過關鍵字精準查詢，訊息也容易被刷屏洗版。此外，由於缺乏結構化的交易狀態管理，經常發生賣家書籍已售出卻未更新或刪文，以及買賣雙方在約定面交時遭受惡意放鳥卻無法處置的情形。

本專案希望針對校園二手書的刊登與面交情境，改善資訊不對稱與搜尋效率，透過強制結構化欄位（EX：科目名稱、開課老師、ISBN）確保資料齊全且格式一致，並提供清晰的交易狀態流轉與放鳥停權機制。
本期不處理：真實線上刷卡或Line Pay金流、超商跨校物流、買賣雙方即時線上聊天室功能。
```

#### 2. 目標角色

| 角色 | 主要任務 | 主要痛點 |
| ----- | -------- | -------- |
| 想省錢的買家學生 | 透過「科目名稱」、「開課老師」或「ISBN」精準查詢所需教科書、查看書況照片、向賣家送出交易邀約 | 現有Line社群訊息洗版較快，並且同學刊登格式有所不同，導致極難以關鍵字搜尋，只能盲目查找訊息 |
| 想清書櫃的賣家學生 | 依系統規範的欄位（科目名稱、開課老師、書籍版本、書況、價格、預期面交地點）上架二手書、管理自己的書籍狀態（上架中、已預訂、已售出） | 每次發文都要自己打字想內容，容易漏寫重要資訊，導致反覆被私訊詢問；書籍售出後也懶得刪文，進而持續收到過期詢問 |
| 平台系統管理人員（小組成員） | 檢視被檢舉的違規商品（EX：非教科書之雜物）、處理面交惡意放鳥之學生停權申訴 | 傳統Line社群缺乏管理工具，無法有效懲罰黑名單，也無法過濾垃圾訊息 |

#### 3. 第一個可操作切片（First Working Slice）

```
本專案第一個可操作切片是一個免安裝資料庫、直接用瀏覽器就能開的網頁原型（利用HTML/CSS/JS加上LocalStorage來模擬資料儲存）。
這個原型主要實作了以下幾個核心功能和操作流程：
1. 買家搜尋與預約：可輸入關鍵字、授課老師、ISBN或價格區間來找書，搜尋結果會即時更新。
2. 防重複預約鎖定：模擬了兩個買家在1秒內預約同一本書的情形，第一個人預約成功後會鎖定書籍狀態，第二個人點擊時會被系統阻斷並跳出安全鎖定提示，防範重複交易。
3. 賣家一頁式快速上架：賣家只要在同一個頁面填完科目、老師、ISBN、書況和面交地點，送出後不用換頁，書籍就會立刻出現在賣家自己的管理清單和買家搜尋頁面中。
4. 放鳥停權與申訴處理：買賣雙方如果被放鳥可以回報，累計滿2次的人帳號會自動被系統停權；被停權的使用者能提交申訴理由，由系統管理員在後台手動審核恢復權限。
```

#### 4. 本期範圍內項目（In-Scope）

1. 結構化二手書上架：賣家上架時必須填寫標準欄位（科目名稱、開課老師、書籍版本、ISBN、書況、價格、預期面交地點）並上傳書籍照片。
2. 多條件精準搜尋篩選：買家可透過關鍵字（EX：科目名稱、開課老師、ISBN）進行模糊查詢，並可依價格區間進行篩選。
3. 面交邀約流程：買家選定書籍後點擊「預約購買」，系統會將書籍狀態切換為「已預約」，並自動產生面交確認單。
4. 面交進度狀態管理：提供雙方確認介面，由賣家於現場面交收現後點擊「確認完成交易」，書籍狀態正式流轉為「已售出」。
5. 違規放鳥檢舉與人工申訴審核：如若面交遭受放鳥，買賣任一方可於系統按下「放鳥回報」，經累計達2次者系統自動予以停權處分；遭停權者可提出申訴，由平台系統管理人員進行人工審核，避免誤判。

#### 5. 本期範圍外項目（Out-of-Scope）

1. 第三方線上金流支付系統：為了在5週內收斂，系統不經手任何金流，一律採現場面交時「現金結帳」或當面轉帳。
2. 物流寄送與店到店系統：本期僅支援校園實體面交（EX：商管大樓、圖書館前），不處理郵寄或超商代收付。
3. 即時線上聊天室：不開發像蝦皮那樣的即時通訊功能，雙方聯繫一律透過系統內建的面交時間地點約定欄位。
4. ISBN全球資料庫API串接：不串接外部圖書館資料庫來自動抓取書名，ISBN欄位在本期純作為資料庫內的比對與搜尋索引。
5. 交易爭議退款仲裁機制：面交過程中若買賣雙方發生糾紛（EX：金額爭議、物品交付爭議），系統不介入處理，僅提供放鳥次數統計、檢舉與停權申訴功能。

#### 8. 程式碼代理不可自行決定的事項

1. 系統架構與前端框架的選用：不得私自決定將專案轉換為React、Next.js或Vue等需要編譯的環境，必須維持純網頁HTML5+CSS+PureJS的單頁面架構。
2. 外部第三方API串接：不可擅自決定引入Firebase、真實資料庫（EX：MySQL/MongoDB）或全球ISBN圖書庫API。
3. 業務邏輯閾值與機制：EX：「放鳥累計2次自動停權」之規則，以及「並行預約衝突阻斷」的邏輯，不可自行修改規則或簡化。
4. 使用者模擬控制項：左側控制台的「當前登入者資訊」與「雙重預約衝突測試」為演示與驗收原型之最重要入口，不可擅自決定刪除、修改或移出畫面。

請依照以下系統分析與設計資料，建立第一個可操作切片。

專案背景：
對於每學期有買賣二手書需求的校內學生而言，目前依賴的Line交易社群缺乏統一格式，同學刊登的書名、版本、授課老師等資訊往往因人而異（EX：同一本資訊系統分析書名有人打英文、有人打中文縮寫），導致買家學生極難透過關鍵字精準查詢，訊息也容易被刷屏洗版。此外，由於缺乏結構化的交易狀態管理，經常發生賣家書籍已售出卻未更新或刪文，以及買賣雙方在約定面交時遭受惡意放鳥卻無法處置的情形。
本專案希望針對校園二手書的刊登與面交情境，改善資訊不對稱與搜尋效率，透過強制結構化欄位（EX：科目名稱、開課老師、ISBN）確保資料齊全且格式一致，並提供清晰的交易狀態流轉與放鳥停權機制。本期不處理真實線上刷卡或Line Pay金流、超商跨校物流、買賣雙方即時線上聊天室功能。

目標角色：
1. 想省錢的買家學生：透過「科目名稱」、「開課老師」或「ISBN」精準查詢所需教科書、查看書況照片、向賣家送出交易邀約。痛點是Line社群訊息洗版快、刊登格式不一、難以關鍵字搜尋。
2. 想清書櫃的賣家學生：依系統規範的欄位（科目名稱、開課老師、書籍版本、書況、價格、預期面交地點）上架二手書、管理自己的書籍狀態（上架中、已預訂、已售出）。痛點是發文容易漏寫重要資訊，書籍售出後懶得更新。
3. 平台系統管理人員（小組成員）：檢視被檢舉的違規商品、處理面交惡意放鳥之學生停權申訴。痛點是傳統Line社群缺乏管理工具，無法有效懲罰黑名單與過濾垃圾訊息。

第一個可操作切片：
本專案第一個可操作切片是一個免安裝資料庫、直接用瀏覽器就能開的網頁原型（利用HTML/CSS/JS加上LocalStorage來模擬資料儲存）。此原型主要實作了以下核心功能：
1. 買家搜尋與預約：可輸入關鍵字、授課老師、ISBN或價格區間來找書，搜尋結果即時更新。
2. 防重複預約鎖定：模擬兩個買家在1秒內預約同一本書的情形，第一個人預約成功後會鎖定書籍狀態，第二個人點擊時會被系統阻斷並跳出安全鎖定提示，防範重複交易。
3. 賣家一頁式快速上架：賣家在同一個頁面填寫書籍科目、老師、ISBN、書況、價格與面交地點，送出後免換頁，書籍即時出現在賣家自己的管理清單與買家搜尋頁面中。
4. 放鳥停權與申訴處理：買賣雙方被放鳥可回報，累計滿2次自動停權且限制交易功能。被停權者能提交申訴，由系統管理員在後台手動審核恢復權限。

本次要支援的需求編號：
FR-01、FR-02、FR-03、NFR-01

本期範圍內：
1. 結構化二手書上架：賣家上架時必須填寫標準欄位（科目名稱、開課老師、書籍版本、ISBN、書況、價格、預期面交地點）並上傳書籍照片。
2. 多條件精準搜尋篩選：買家可透過關鍵字（EX：科目名稱、開課老師、ISBN）進行模糊查詢，並可依價格區間進行篩選。
3. 面交邀約流程：買家選定書籍後點擊「預約購買」，系統會將書籍狀態切換為「已預約」，並自動產生面交確認單。
4. 面交進度狀態管理：提供雙方確認介面，由賣家於現場面交收現後點擊「確認完成交易」，書籍狀態正式流轉為「已售出」。
5. 違規放鳥檢舉與人工申訴審核：如若面交遭受放鳥，買賣任一方可於系統按下「放鳥回報」，經累計達2次者系統自動予以停權處分；遭停權者可提出申訴，由平台系統管理人員進行人工審核，避免誤判。

本期範圍外：
1. 第三方線上金流支付系統：不經手任何金流，一律採現場面交時「現金結帳」或當面轉帳。
2. 物流寄送與店到店系統：僅支援校園實體面交（EX：商管大樓、圖書館前），不處理郵寄或超商代收付。
3. 即時線上聊天室：不開發像蝦皮那樣的即時通訊功能，雙方聯繫一律透過系統內建的面交時間地點約定欄位。

5. 交易爭議退款仲裁機制：面交過程中若買賣雙方發生糾紛，系統不介入處理，僅提供放鳥次數統計、檢舉與停權申訴功能。

技術限制：
1. 前端技術：HTML5、CSS3(Vanilla CSS)、Vanilla JavaScript(ES6+)、Google Fonts(Inter/Noto Sans TC)。

驗收條件：
