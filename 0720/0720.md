### F. 使用案例描述

```
使用案例編號：UC-02
使用案例名稱：預約面交購買

目標：買家對有興趣的二手教科書發起預約購買，鎖定書籍狀態以防止他人重複預約，並產生面交單。
主要參與者：買家學生
支援參與者：賣家學生
觸發事件：買家在精準搜尋列表中找到所需教科書，點擊「預約購買」按鈕。

前置條件：
1. 該書籍在系統中的狀態為「上架中」。
2. 買家的帳號狀態為「正常」（未因放鳥達2次而被停權）。

成功後置條件：
1. 該書籍在系統中的狀態變更為「已預約」。
2. 產生一筆交易媒合紀錄，包含買賣雙方學號、書籍資訊及面交地點。
3. 買家與賣家端即時顯示該筆面交確認資訊，且該書籍於商場中不可被其他人點選預約（防重複鎖定）。

最低保證：若預約失敗，書籍狀態維持「上架中」，系統提示失敗原因且不扣除或修改任何原有交易紀錄。

主要流程：
1. 買家在搜尋結果中點選所需書籍卡片。
2. 買家點擊「預約購買」按鈕。
3. 系統檢查買家帳號狀態，確認為正常。
4. 系統檢查該書籍狀態，確認為上架中。
5. 系統執行併行狀態鎖定，將書籍狀態修改為「已預約」，並建立面交交易訂單。
6. 系統彈出面交確認單Modal，顯示雙方聯絡學號與預期面交地點。
7. 系統將此預約紀錄寫入日誌Console。

替代流程：
- 編號：ALT-02-01（併行衝突）
- 分支點：主要流程第5步
- 條件：在買家A點擊預約的幾乎同一時間，買家C也對同一本書點擊預約，且買家C些微落後買家A。
- 步驟：
  1. 系統檢查發現該書已被買家A搶先鎖定。
  2. 系統阻斷買家C的請求，彈出「系統安全鎖定：此書籍已被預約」之警告提示。
  3. 書籍按鈕即時變更為禁用狀態「已預約」。
- 結束或返回位置：買家C點擊確認關閉提示，返回搜尋頁面。

例外流程：
- 編號：EXC-02-01（帳號停權限制）
- 分支點：主要流程第3步
- 條件：買家學生的帳號狀態為「已停權」（放鳥次數達2次）。
- 步驟：
  1. 系統阻斷預約操作，頁面頂部顯示停權警告Banner。
  2. 系統提示「您的帳號已被停權，無法進行交易預約，請至左側提交申訴」。
- 結束狀態：不產生交易紀錄，返回搜尋頁面。

業務規則：
1. 一本書同時僅能被一個買家預約（BR-01）。
2. 已預約之書籍在商場列表中必須顯示「已預約（面交中）」且按鈕為Disabled。

輸入資料：買家學號、書籍ID、面交約定地點。
輸出資料：面交確認單資訊（雙方學號、面交地點、時間）、系統交易衝突日誌。
狀態變化：書籍狀態「上架中」 -> 「已預約」；新增交易訂單狀態為「待面交」。
對應需求：FR-01, NFR-01
對應驗收條件：AC-02-01, AC-02-02
待確認問題：若買家預約後欲取消，是否有冷卻時間？（本期暫不支援主動取消）
```

```
使用案例編號：UC-05
使用案例名稱：放鳥停權與申訴處理

目標：當面交遭遇放鳥時，受害者可向系統申報；系統累計放鳥次數滿2次將自動停權；被停權者可提交申訴，由管理員人工覆核解鎖。
主要參與者：買家學生/賣家學生（回報/申訴者）、平台系統管理人員（審核者）
支援參與者：無
觸發事件：交易狀態為「已預約」之訂單逾期未面交，且對手無故未到（放鳥）。

前置條件：存在一筆狀態為「已預約」的面交訂單。

成功後置條件：
1. 違規者放鳥次數累計加1。
2. 若違規者放鳥次數累計達2次，帳號狀態自動更新為「已停權」，並顯示停權申訴表單。
3. 管理員於後台核准申訴後，違規者帳號恢復為「正常」，且放鳥次數重置歸零。

最低保證：回報放鳥時必須輸入具體原因，否則系統拒絕受理，以防惡意濫用。

主要流程：
1. 遭受放鳥的使用者點選預約紀錄卡片上的「放鳥回報」按鈕。
2. 系統跳出輸入框，要求輸入放鳥詳情原因。
3. 使用者填寫並送出。
4. 系統累計違規者之放鳥次數，若次數達2次，自動將其使用者狀態改為「已停權」。
5. 被停權之使用者登入時，系統於個人狀態卡顯示「已停權」，並於側邊欄渲染「帳號遭停權申訴」表單。
6. 被停權者填寫文字理由並點擊「送出人工申訴」。
7. 管理人員登入後，進入管理後台之「申訴審理」分頁，查閱該學生的申訴與放鳥歷史。
8. 管理人員點選「核准（恢復權限）」。
9. 系統將該學生放鳥次數歸零，狀態重設為正常，並自申訴列表中移除該案。

替代流程：
- 編號：ALT-05-01（管理員駁回申訴）
- 分支點：主要流程第8步
- 條件：管理人員認為申訴理由不合理。
- 步驟：
  1. 管理人員點選「駁回申訴」。
  2. 該申訴案件結案。
  3. 學生帳號維持「已停權」狀態。
- 結束或返回位置：管理人員返回申訴審理頁面。

例外流程：
- 編號：EXC-05-01（放鳥回報防呆）
- 分支點：主要流程第3步
- 條件：回報原因為空白。
- 步驟：
  1. 系統跳出警告Toast「回報失敗！請填寫詳細放鳥原因」。
- 結束狀態：不增加放鳥次數。

業務規則：
1. 放鳥次數累計滿2次自動停權帳號（BR-02）。
2. 停權狀態下禁止進行搜尋預約與上架管理交易（BR-03）。

輸入資料：違規者學號、放鳥詳情文字、申訴理由文字。
輸出資料：放鳥回報單、申訴處理結果、更新後的使用者狀態。
狀態變化：使用者帳號「正常」 -> 「已停權」 -> 「正常」；放鳥次數累計 -> 歸零。
對應需求：FR-03
對應驗收條件：AC-04-01, AC-05-01
待確認問題：是否應限制申訴提交的次數？（本期暫不限制）
```
