### B. 7/6 專案章程修正摘要

| 項目 | 原本內容 | 修正後內容 | 修正理由 |
| ---------- | ---------- | ---------- | ---------- |
| 專案題目 | 診所候診叫號與現場預約系統 |            |            |
| 問題陳述 | 小型診所目前的候診叫號流程仍仰賴人工作業，病患報到後不知道還要等多久，只能在候診區乾等或反覆詢問櫃檯；櫃檯人員需手動翻紙本、口頭喊號，遇到插隊或爽約時難以妥善應對；醫生也無法即時掌握候診區的等候狀況；診所管理人員則因缺乏看診人數、爽約次數等數據，難以檢討流程或調整人力安排。尤其在尖峰時段，這些問題更容易導致現場秩序混亂，進而使病患抱怨增加以及診所評價下降。<br /><br />我們希望針對病患現場報到後的候診情境，改善候診透明度與現場管理效率，讓病患能自行查詢叫號進度，櫃檯人員也能更有效率地進行作業與管理。<br />本期不處理：健保系統或電子病歷的資料串接、跨診所的醫生班表管理、線上遠端事先掛號。 |            |            |
| 範圍內項目 | 1.病患可透過看診號查詢目前叫號進度。<br />2.櫃檯人員可將現場報到的病患加入候診隊列，並記錄看診醫生與預估看診時間。<br />3.櫃檯人員可調整隊列順序，插入急診病患優先看診，或將爽約病患標記並移出目前隊列。<br />4.系統會根據當下候診隊列，即時更新狀態。<br />5.診所管理人員可檢視每日看診人數、爽約次數等基本統計數據。 |            |            |
| 範圍外項目 | 1.與健保系統或電子病歷的資料串接。<br />2.線上遠端事先掛號。<br />3.看診內容記錄、病歷撰寫或處方開立等醫療相關功能。<br />4.病患身分驗證或會員帳號系統。<br />5.跨診所或跨診間的醫生班表管理。 |            |            |
| 原型核心任務 | 1. 病患加入候診隊列<br />2. 查詢叫號進度<br />3. 調整隊列<br />4. 檢視每日候診統計 |            |            |

### E. 可行性分析

| 可行性類型 | 分析內容 | 判斷結果 | 需要調整的範圍 |
| ---------- | ---------- | ---------- | ---------- |
| 技術可行性（technical feasibility） | 系統核心功能為隊列的新增、查詢、狀態更新，技術難度不高，只要小組成員具備基礎程式開發能力即可應付 | 可行 | 如若原規劃有即時更新功能，建議簡化為手動刷新或定時輪詢，避免技術複雜度超出小組能力 |
| 操作可行性（operational feasibility） | 櫃檯人員需改變原本手寫紙本、口頭喊號的作業習慣，改用系統操作；病患需學習用看診號查詢進度 | 大致可行，但需簡化操作介面 | 介面設計力求簡單，並且需要提供簡短的操作說明或教學，降低使用門檻 |
| 時程可行性（schedule feasibility） | 專案時程僅5週，需完成需求分析、設計、開發、測試等階段 | 可行，但範圍需控制 | 已排除多醫生多診間、健保串接、線上預約等功能，需持續把關範圍不再擴增 |
| 經濟可行性（economic feasibility） | 專案為課程原型開發，不涉及實際採購或營運成本，僅需考量開發時間與人力投入 | 可行 | 需留意小組成員時間分配是否足夠支撐開發進度 |
