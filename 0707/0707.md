### B. 7/6 專案章程修正摘要

| 項目 | 原本內容 | 修正後內容 | 修正理由 |
| ----- | -------- | ------------ | -------- |
| 專案題目 | 基於結構化查詢的校園二手教科書媒合與面交管理平台 |          |          |
| 問題陳述 | 對於每學期有買賣二手書需求的校內學生而言，目前依賴的Line交易社群缺乏統一格式，同學刊登的書名、版本、授課老師等資訊往往因人而異（EX：同一本資訊系統分析書名有人打英文、有人打中文縮寫），導致買家學生極難透過關鍵字精準查詢，訊息也容易被刷屏洗版。此外，由於缺乏結構化的交易狀態管理，經常發生賣家書籍已售出卻未更新或刪文，以及買賣雙方在約定面交時遭受惡意放鳥卻無法處置的情形。<br><br>本專案希望針對校園二手書的刊登與面交情境，改善資訊不對稱與搜尋效率，透過強制結構化欄位（EX：科目名稱、開課老師、ISBN）確保資料齊全且格式一致，並提供清晰的交易狀態流轉與放鳥停權機制。<br>本期不處理：真實線上刷卡或Line Pay金流、超商跨校物流、買賣雙方即時線上聊天室功能。 |          |          |
| 範圍內項目 | 1. 結構化二手書上架：賣家上架時必須填寫標準欄位（科目名稱、開課老師、書籍版本、ISBN、書況、價格、預期面交地點）並上傳書籍照片。<br>2. 多條件精準搜尋篩選：買家可透過關鍵字（EX：科目名稱、開課老師、ISBN）進行模糊查詢，並可依價格區間進行篩選。<br>3. 面交邀約流程：買家選定書籍後點擊「預約購買」，系統會將書籍狀態切換為「已預約」，並自動產生面交確認單。<br>4. 面交進度狀態管理：提供雙方確認介面，由賣家於現場面交收現後點擊「確認完成交易」，書籍狀態正式流轉為「已售出」。<br>5. 違規放鳥檢舉停權：如若面交遭受放鳥，買賣任一方可於系統按下「放鳥回報」，經累計達2次者，系統自動予以停權處分。 |          |          |
| 範圍外項目   | 1. 第三方線上金流支付系統：為了在 5 週內收斂，系統不經手任何金流，一律採現場面交時「現金結帳」或當面轉帳。<br>2. 物流寄送與店到店系統：本期僅支援校園實體面交（EX：商管大樓、圖書館前），不處理郵寄或超商代收付。<br>3. 即時線上聊天室：不開發像蝦皮那樣的即時通訊功能，雙方聯繫一律透過系統內建的面交時間地點約定欄位。<br>4. ISBN全球資料庫PI串接：不串接外部圖書館資料庫來自動抓取書名，ISBN欄位在本期純作為資料庫內的比對與搜尋索引。<br>5. 非教科書之一般雜物市集管理：嚴格限制商品分類，本期不處理衣服、電子產品等一般校園二手物買賣。 |          |          |
| 原型核心任務 | 1. 賣家結構化刊登與買家精準搜尋：賣家填寫標準欄位與上傳照片並送出上架。買家透過關鍵字進行查詢，並成功查詢到相關資訊。<br>2. 面交邀約流程與鎖定書籍狀態：買家看到需要的教科書，點擊「預約購買」，並選擇期望的面交地點。系統成功建立媒合訂單，且該書籍在首頁的狀態自動變為「已預約」，防止其他人重複預約。<br>3. 買賣雙方在校園見面，買家檢查書況並給予現金後，賣家打開系統點擊「確認完成交易」。系統將訂單結案，書籍狀態切換為「已售出」，並將該紀錄移入兩人的歷史交易檔案中。 |          |          |

### D. 權力 / 關注矩陣

| 類型 | 本組利害關係人 | 後續需求蒐集策略 |
| ----- | ------------ | --------------- |
| 高權力、高關注 | 平台系統管理人員 | 優先設計管理後台的「檢舉審核」與「停權一鍵處分」介面，確保系統規則具備強制力 |
| 高權力、低關注 | 學校教務處／資訊處 | 採取「主動宣告」策略。在專案章程與原型首頁明確標示「本平台不涉及線上金流、不存在敏感個資、僅限校內面交」，消除校方資安疑慮，避免被限制與推翻 |
| 低權力、高關注 | 買家學生／賣家學生 | 列為首要深度訪談與問卷對象。針對同儕進行訪談，蒐集其在Line社群買賣最常遺漏的欄位（EX：科目名稱、書籍版本），作為結構化欄位設計的依據 |
| 低權力、低關注 | 開課授課老師／校園周邊實體二手書店 | 保持基本關注即可。可於系統欄位中加入「ISBN」與「版本」之強制驗證，間接滿足老師對於教學版本正確的需求 |

### E. 可行性分析

| 可行性類型 | 分析內容 | 判斷結果 | 需要調整的範圍 |
| ------------ | -------- | -------- | ------------ |
| 技術可行性（technical feasibility） | 本專案核心為資料的「結構化輸入、模糊查詢、與狀態流轉（上架->已預約->已售出）」。使用Figma原型工具搭配元件狀態切換即可100%完美模擬此商務邏輯，不需撰寫後端真實代碼，開發工具與技術支援度高 | 可行 | 為了防範技術風險，「即時聊天室」與「串接外部全球ISBN API自動帶出書名」確定不可行，需全面改由系統內建「面交約定欄位」與「手動輸入關鍵字」來替代 |
| 操作可行性（operational feasibility） | 買賣雙方皆為校內同儕，具備極高的高擬真同理心與使用情境。系統介面設計只要比Line社群更好搜尋、上架欄位引導明確，使用者的使用意願將會非常高，流程變革阻力極低 | 可行 | 需加入「誠信評價機制」，否則單純的媒合無法解決實體面交最常發生的放鳥操作痛點 |
| 時程可行性（schedule feasibility） | 暑修時程僅有5週，時間極度緊迫。本專案完全聚焦於「書籍上架、精準搜尋、面交確認」三個最核心的黃金流程，去除了所有旁支功能，安排可5週內完成擬真原型 | 可行 | 為了控制工作量，「跨校物流」與「退換貨爭議仲裁流程」確定不可行，需移至範圍外 |
| 經濟可行性（economic feasibility） | 本專案在暑修期間純粹進行系統分析、設計與互動原型展示，使用的繪圖與建模工具皆為免費版本（Figma／Draw.io／ GitHub），不需要任何伺服器部署、資料庫付費或授權成本，成本效益合理 | 可行 | 成本為零，不需要進行調整 |
