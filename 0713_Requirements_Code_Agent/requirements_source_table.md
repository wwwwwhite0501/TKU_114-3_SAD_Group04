#### 1. 專案背景

淡江大學學生每學期皆有龐大的二手教科書買賣需求，然而現有的 Line 交易社群缺乏規範，同學刊登的格式五花八門、代稱不一，造成買家透過關鍵字搜尋不到對應教材。本專案開發一套結構化媒合平台，強制標準化欄位輸入，實現校園二手書的高效、誠信流通。

#### 2. 目標角色

| 角色               | 主要任務                               | 主要痛點                                                           |
| ------------------ | -------------------------------------- | ------------------------------------------------------------------ |
| 想省錢的買家學生   | 依老師姓名精準查詢課本、發起面交約定。 | 社群洗版快，且因為書名雜亂，用關鍵字根本搜尋不到要買的必修課本。   |
| 想清書櫃的賣家學生 | 規範化刊登書籍、現場確認收錢結案。     | 每次發文都要手打一堆資訊很麻煩；書賣掉後懶得刪文會狂被舊訊息打擾。 |

#### 3. 第一個可操作切片（First Working Slice）

本次實作只聚焦完成一條「黃金端到端流程」：賣家輸入科目與老師姓名成功上架一本書 -> 買家在首頁透過輸入授課老師姓名精準搜尋到該本書 -> 買家點擊預約購買並選擇面交地標 -> 書籍狀態流轉為「已預約」且按鈕變灰阻擋。

#### 4. 本期範圍內項目（In-Scope）

1. 單頁式前端網頁介面（包含角色切換面板、書籍上架表單、書籍列表、搜尋篩選列）。
2. 書籍物件的四大基礎狀態控制（上架中、已預約、已售出、停權中）。
3. 前端 JavaScript 模擬陣列資料庫（使用 Mock Data 進行新增、過濾查詢、狀態變更）。
4. 「放鳥檢舉」與累積 2 次觸發前端模擬停權的操作按鈕。

#### 5. 本期範圍外項目（Out-of-Scope）

1. 真實後端伺服器與 SQL 資料庫部署。
2. 任何線上刷卡、Line Pay、街口支付等金流串接（統一改為按鈕點擊「模擬現場收現」）。
3. 買賣雙方即時線上聊天對話室 Chatroom。
4. 串接外部圖書館或全球 ISBN 的 API 自動帶出書名功能。

#### 8. 程式碼代理不可自行決定的事項

1. 不得自行建立「購物車」或「多商品一次結帳」的複雜流程。
2. 不得自行引入 Firebase 或 LocalStorage 來做資料持久化，一律使用記憶體變數。
3. 不得自行捏造「開課系統串接」、「選課系統連動」等超出範圍的虛擬背景。
